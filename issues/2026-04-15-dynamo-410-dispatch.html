Features

The Workspace Finally Remembers You

Features

Dynamo's workspace has always been stateless in frustrating ways — panels reset to defaults every session, menus were buried in submenus, and navigating a large graph meant scrolling. 4.1.0 changes the contract: the Extensions panel now remembers whether you minimized it, so your layout survives restarts. Home and End jump to the leftmost and rightmost nodes on the canvas without scrolling. The Extensions menu is promoted to sit alongside every other top menu. And deleting an inline Watch node no longer orphans connected ports — wires now rewire cleanly through the deleted node.

Features — Continued

The API Sheds a Dependency It Never Needed

Features

External node libraries that need to declare their menu category have always been forced to reference DynamoCore — a heavy package — just to access NodeCategoryAttribute. In 4.1.0 that attribute is type-forwarded to DynamoServices, a lighter package. Libraries can now drop the heavyweight reference. ActionGroupDisposable is also made nestable: complex multi-step operations can now compose their undo groups cleanly without conflicts. And Dynamo's graph file format gets a comprehensive JSON Schema and Markdown specification for the first time — a machine-readable contract for tools that parse .DYN files.

Bug Fixes — Part I

Six Pin Bugs Walk Into a Release. None Walk Out.

Bug Fixes

Connector pins are the anchors you attach to wires to lock them in place during graph edits. They live in the graph file. Six separate ways this system could fail were fixed in one release: pins shifted upward after save/reload, disappeared entirely when using Save As, were silently dropped from clipboard copy/paste, couldn't be undone with Ctrl+Z, notes lost their pin when the note was deleted, and rapid drag operations would lose the pin through a transient reconnection path. The fixes cover both serialization (pins now survive all save paths) and the interaction model (undo stack tracks all pin operations correctly).

6
Connector Pin Bugs Fixed in One Release
Pins · Clipboard · Undo
What This Means for You
If you've been working around pin loss on Save As, or rebuilding pins after reload drift, those workarounds can be dropped. The transient-pin reconnection hardening is the most subtle fix — it closes a race condition that surfaces specifically during rapid editing, which matters for automation-driven or scripted graph construction workflows.
03
Bug Fixes — Part II

The Camera Drifts. The Gizmo Fails. Both Fixed.

Bug Fixes

Two viewport failures shared the same root cause: the transform gizmo stopped redrawing correctly during drag operations, and the camera slowly drifted when zoomed in. Both traced back to a WPF BitmapCache shared across rendering threads — a race condition that silently corrupted render state. Pan and zoom crashes on large graphs followed the same pattern. The second batch of fixes also addresses premature geometry garbage collection: CLR containers holding LibG geometry pointers were sometimes freed before downstream nodes finished reading them, causing silent null returns or unpredictable crashes. The fix ties geometry lifetimes to the nodes that own them.

26
Total Bugs Closed in 4.1
WPF · BitmapCache · Geometry GC
What This Means for You
The geometry GC fix matters if you have nodes that hold LibG geometry references across async or multi-step execution — the previous behavior could silently return null or crash downstream. The shared BitmapCache pattern it replaces is worth checking in your own WPF components if you render geometry previews or use threaded rendering anywhere in custom view extensions.

Geometry & LibG

Curves That Finally Do What They Promise

Geometry & LibG Updates

Voronoi diagrams partition space by "which seed point is nearest?" — Dynamo's implementation failed when edge lengths varied widely, producing broken output on realistic inputs. A relative edge-length filter now handles those cases. PolyCurve.ByPoints had undefined behavior at boundary conditions: it could return different curve types depending on subtle input differences. The return type is now unified and predictable. Periodic NURBS curves — closed smooth curves like circles — gain new API exposure for their knot and control point topology, filling a gap that previously required verbose DesignScript workarounds. Curve.PullOntoPlane and PullOntoSurface now return concrete curve types with performance improvements.

What This Means for You
The new periodic NURBS APIs (PR #16896) expose curve topology that previously required DesignScript workarounds. If your C# nodes wrap LibG geometry, the unified PolyCurve.ByPoints return type and concrete Curve.PullOnto* types may change downstream type checks — worth a regression pass on affected nodes before shipping the upgrade to users.

Performance

One Low-Level Fix. Massive Speed Recovery.

Performance Improvements

Dynamo keeps a hash table to locate geometry objects quickly. Every object needs a unique key. The old method borrowed keys from .NET's default GetHashCode() — which is allowed to return the same value for two different live objects. When that happens in a large table, every lookup degrades from an instant O(1) jump to a slow linear search, and the problem compounds with more objects. The fix is one line: use memory addresses as hash keys instead. Memory addresses are guaranteed unique for live objects, eliminating the collision class entirely. A second fix stops expensive view model calculations from blocking the WPF rendering thread by deferring them to the Dispatcher.

0
Geometry Hash Collisions After Fix
CLR · Dispatcher · LibG
#16754 — Identity hash codes eliminate the geometry marshaling collision cascade
What This Means for You
PR #16754 is the sleeper hit of this release. If your graphs slow to a crawl with hundreds of geometry objects in play, this hash fix alone justifies the upgrade. The Dispatcher deferral pattern in the view model fix is also a solid technique to adopt in any WPF performance work you're doing in the broader Dynamo codebase — defer, don't block.

PythonNet3

Purple Is Gone. PythonNet3 Is the Baseline.

PythonNet3

When Dynamo migrated Python nodes from IronPython to PythonNet3, the team added a distinctive purple syntax highlight as a visual signal: "this node is running the new runtime." That marker served its purpose during the migration period. With PythonNet3 now the only supported runtime, the purple highlight is redundant and removed — Python nodes look like everything else. The bundled sample scripts have been updated throughout, replacing all PythonNet2-style import patterns. Python in Dynamo is no longer a migration story. It's just the baseline.

0
Purple Migration Flags Remaining
PythonNet3 · Python · IronPython
#16615 — Purple syntax highlighting removed; PythonNet3 is now the unambiguous default
What This Means for You
If you maintain Python node documentation or sample scripts for Dynamo users, now is the time to audit for remaining PythonNet2-style imports (PR #16600) and update any screenshots that show the old purple highlight. For users, this is invisible — for documentation maintainers, it's a content refresh moment.

Build & Infrastructure

The Pipeline Catches Up to .NET 10

Build & Infrastructure

After RC 4.1.0 was cut, master was bumped to 4.2.0 and the performance test suite was upgraded to .NET 10. The SDK roll-forward policy in global.json was relaxed: instead of requiring an exact SDK version match, builds now accept any newer patch. This means a .NET SDK patch update no longer silently breaks local builds or CI. The npm registry reachability check was rewritten to use a reliable WebClient call after the previous Invoke-WebRequest approach produced intermittent failures in CI environments. These changes harden the developer loop for everyone contributing to or forking the Dynamo repo.

0
Builds Broken by SDK Patch Updates
.NET 10 · CI · NuGet
What This Means for You
The rollForward change in global.json means your local builds now accept .NET 10 patch updates without a manual SDK pin edit. If you maintain CI scripts for Dynamo forks or downstream repos, the Invoke-WebRequest → WebClient pattern replacement (PR #16834) is worth reviewing — the brittle pattern it replaces may exist in your own pipeline scripts.

AI Agent Skills & Tooling

Skills Get a Canonical Home and a Validation Gate

AI Agent Skills & Tooling

Agent skills in the Dynamo repo were scattered across tool-specific directories — each IDE or AI assistant dropped its configuration wherever it expected. This release canonicalizes everything under .agents/skills and adds a validate-skills CI check that fails the build if any tool's directory drifts from that path. Seven new skills were added: dynamo-unit-testing for structured C# test generation, a C# Janitor for automated code health cleanup, UX Designer and Content Designer agents aligned to the Autodesk Weave Design System, plus three experimental architecture agents. The AI tooling surface in the repo is now consistent across Claude Code, Cursor, and GitHub Copilot.

The validate-skills CI check enforces consistency across IDE wrappers — if you contribute using Claude Code, Cursor, or VS Copilot, .agents/skills is now your canonical source of truth. The dynamo-unit-testing and C# Janitor agents are immediately applicable to day-to-day C# work on the Dynamo repo. Spin up the Janitor on any module you're refactoring.
